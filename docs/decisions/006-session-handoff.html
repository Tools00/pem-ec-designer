Session-Übergabe · ohne Kontext-Verlust

Wie du in einer neuen Session weitermachst

Drei Mechanismen sorgen dafür, dass eine frische Claude-Session sofort weiß, wo wir stehen. Plus: was Fork/Branch dir bringt — und was nicht.

Die 3 Persistenz-Säulen

📖

CLAUDE.md im Repo

primär

Bootstrap-Manual. Jede neue Claude-Session liest das automatisch beim Start. Enthält: aktueller Stand, offene Pfade, Verhaltensregeln, Bootstrap-Sequenz.

# Schon aktualisiert in dieser Session: $ head -20 CLAUDE.md
📋

ADRs + Decision-HTML

strukturell

docs/adr/*.md = warum so entschieden wurde · docs/decisions/*.html = die interaktiven Entscheidungs-Karten als Beleg. Roadmap zeigt Status visuell.

$ ls docs/adr/ docs/decisions/ $ open docs/decisions/000-roadmap.html
⛓

Git-Historie + CHANGELOG

technisch

Commit-Messages erzählen was, CHANGELOG.md warum. Conventional Commits machen die History grep-bar (feat:, ci:, fix:).

$ git log --oneline -10 $ cat CHANGELOG.md
🤖

CI als externes Gedächtnis

automatisiert

Tests + Schema-Drift laufen bei jedem Push. Wenn was kaputt ist, sieht's die nächste Session sofort. Keine still kaputten Branches.

$ gh run list --limit 3 $ gh run view --log

Bootstrap-Sequenz für die nächste Session

In den Projekt-Ordner wechseln

Claude liest CLAUDE.md automatisch ab dem Moment, wo du in dem Verzeichnis startest.

$ cd path/to/Simulation-tools/pem-ec-designer

Claude-Session öffnen — neu oder fortgesetzt

--continue nimmt die letzte Session in diesem Ordner mit Inhalt mit. claude ohne Flag startet frisch (CLAUDE.md ist genug Bootstrap).

$ claude --continue # mit voller Historie $ claude # frisch, liest CLAUDE.md

Status zeigen lassen

Erste Frage in der neuen Session — Claude reagiert auf CLAUDE.md + Git-State + CI-Stand.

User: "Wo stehen wir? Welche Pfade sind offen?"

Pfad wählen und weitermachen

A · weiter Specs · B · erster Generator · C · UI-Skelett · D · pause. Aus CLAUDE.md.

User: "B" # oder welcher Pfad auch immer

Fork vs. Branch vs. Clone — was bringt dir was?

FeatureWofürHilft Session-Wechsel?Nutzen für dich gerade
Fork Repo eines anderen Users kopieren, um eigene Änderungen vorzuschlagen (Pull Request) nein Du bist Repo-Owner — Fork ist für Contributors auf fremden Repos. Bei dir irrelevant.
Branch Parallele Entwicklung im selben Repo, später per Merge zusammengeführt nein — aber nützlich für Experimente Sinnvoll wenn du parallel zwei Pfade probierst (z.B. experiment/build123d-membrane) ohne main zu brechen.
Clone Repo auf zweitem Gerät bekommen ja — für Multi-Device-Arbeit Wenn du auf einem zweiten Rechner weitermachst: git clone git@github.com:Tools00/pem-ec-designer.git
Tag / Release Versionsstand markieren (v0.1.0, v0.2.0) ja — markiert Meilensteine Setzen wir bei v0.1, wenn der erste Generator läuft. Jetzt noch zu früh.
GitHub Issues Aufgaben/Bugs strukturiert verfolgen ja — externes Backlog Können wir aus den offenen Pfaden in CLAUDE.md generieren. Sinnvoll wenn mehrere Sessions parallel laufen.
GitHub Projects Kanban-Board für Issues ja — visuell Overkill für Solo-Projekt. Erst wenn Co-Maintainer dazukommen.

Empfehlung — Branch-Pattern für Experimente

Sobald wir vom Schema-/Spec-Pfad zu echten Generatoren oder UI-Code übergehen, ist es klug, auf Branches zu arbeiten. Nicht weil es nötig ist, sondern weil main dann immer „grün und reviewbar" bleibt.

# Beispiel: build123d-Generator auf eigenem Branch $ git switch -c experiment/membrane-generator # … arbeiten, committen … $ git push -u origin experiment/membrane-generator # Dann via gh oder Web-UI Pull Request gegen main $ gh pr create --fill

Was schon eingebaut ist · Checkliste

  • CLAUDE.md aktualisiert — keine „leeres Skelett"-Lüge mehr; reflektiert v0.0.1-Stand
  • ADR-001 + ADR-002 committed — Entscheidungen mit Begründung im Repo
  • CHANGELOG.md geführt — Versions-Log ab v0.0.1
  • CI grün auf main — letzte Session hat funktionierenden Stand hinterlassen
  • 5 Decision-HTMLs unter docs/decisions/ — visuelles Klickprotokoll der Wahl
  • 45/45 Tests grün — Sicherheitsnetz für Refactor

Wenn du jetzt aussteigst

Es ist nichts „im Flug". Working tree clean, CI grün, alle Entscheidungen dokumentiert. Die nächste Session — egal ob in 1 Stunde oder 2 Wochen — kann sofort an Pfad A/B/C/D weitermachen, ohne dass irgendwas verloren geht.

Letzter Schritt jetzt: commit + push der aktualisierten CLAUDE.md + diese Handoff-Seite, dann ist der Stand auch auf GitHub gespiegelt.
