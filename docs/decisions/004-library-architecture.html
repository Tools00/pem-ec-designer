Library-Architektur · 6 Entscheidungen

Wie liegt die Library auf Disk? Wie validiert? Wie bindet sie Quellen, Materialien, Units? Empfehlung ist je markiert. Schema- und Tree-Tab oben rechts zeigen das Ergebnis live.

D1

Storage-Layout — wie liegt der Katalog auf Disk?

60+ Items wollen sortiert werden. Diff-Friendly soll's auch sein für Git.

offen
library/components.json — eine Datei
Alle 60+ Komponenten in einer JSON. Pro: simpel · Contra: Merge-Konflikte, schlechte Diffs ab ~30 Items.
library/components/{category}.json — pro Kategoriebalanced
membrane.json, anode_cl.json, cathode_cl.json, gdl.json, bpp.json, endplate.json, gasket.json, flow_field.json. ~8 Files, ~7 Items each.
library/components/{cat}/{id}.yaml — pro Komponente
60+ einzelne Files. Pro: beste Diffs, Plugin-Pattern · Contra: File-Wust, langsamer Bulk-Edit.
SQLite-DBover-engineered
Library als library/components.db. Pro: Queries · Contra: nicht human-editable, schlechte Git-Diffs.
D2

Spec-Validierung — Schema-Layer

Tippfehler in JSON sollen früh fliegen. Static-Analysis im Editor wäre nett.

offen
Pydantic v2 only
Runtime-Validierung beim Laden, IDE-Completion in Python. Kein Editor-Schema in IDE für die JSON-Files.
JSON Schema only
Editor (VS Code) validiert beim Tippen. Kein Pydantic-Layer in Code → manueller Cast.
Beides — Pydantic + auto-generiertes JSON Schematwo birds
Pydantic-Modelle als Source of Truth, model_json_schema() exportiert schema.json für Editoren + CI.
D3

Material vs. Component — separat oder inline?

„Nafion 117" wird in mehreren Komponenten referenziert. DRY oder self-contained?

offen
Material inline in Component
Jede Komponente trägt ihre Material-Eigenschaften selbst. Pro: self-contained · Contra: Update an Nafion-Werten = N Stellen ändern.
Material-Katalog separat, per ReferenceDRY
library/materials.json mit ~20 Materialien. Komponenten referenzieren via material_ref: "nafion-117".
D4

Quellen-Storage — wie sind Paper-Refs gebunden?

Strict-Quellen-Wahl verlangt Paper-Beleg pro Wert. Format?

offen
Volle Citation inline pro Wert
{value: 120, source: "Carmo et al. 2013, IJHE, Table 2"}. Lesbar aber redundant.
Reference-ID + separate BibTeXscientific standard
{value: 120, source: "carmo2013ijhe.tab2"} → resolved via library/sources.bib. Standard für wissenschaftliche Reproduzierbarkeit.
Source-Block pro Komponente, kein per-value
Eine Quellenliste pro Komponente, nicht pro Feld. Contra: verliert Granularität — Tafel-Slope von Paper A, j₀ von Paper B nicht trennbar.
D5

Component-ID-Schema

Wie heißen die Specs? Eindeutigkeit + Lesbarkeit + Sortierbarkeit.

offen
Flat slug — nafion-117
Kurz, aber Kategorie verloren — nafion-117 könnte Membran oder Dichtung sein.
Hierarchical — membrane.nafion.117strukturiert
Sortierbar, machine-parseable, human-readable. Drei Ebenen reichen für 60+ Items.
UUID — c1b8...
Maximal eindeutig, aber unfreundlich. Nur sinnvoll wenn Komponenten user-erstellt werden (nicht der Fall).
D6

Engineering-Units in JSON

Paper geben µm, mil, m durcheinander. Wie speichern wir Werte?

offen
SI only — thickness: 0.000183
Konsistent, aber lesbar nur mit Lupe. thickness: 0.000183 ist 183 µm? oder 0.183 mm? Verschreiber-anfällig.
Suffix — thickness_um: 183
Lesbar, aber kein expliziter Unit-Layer. Mix-Risiko bleibt (jemand schreibt thickness statt thickness_um).
Explicit object — {value: 183, unit: "um"}strict-konform
Pydantic-Validator konvertiert zu SI beim Laden. Kein Verwechslungsrisiko, alle Quellen (µm/mil/m) direkt aus Paper übernehmbar.
