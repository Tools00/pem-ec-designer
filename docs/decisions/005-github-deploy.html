GitHub-Deploy · 5 Stufen

Deploy ist nicht „einmal machen", sondern eine Sequenz mit klaren Gates. Was zu welchem Zeitpunkt passiert — und warum nicht früher.

1
JETZT

Lokaler git init + erster Commit

private (lokal)

Repo lokal initialisieren, sauberer Baseline-Commit mit ADR-001/002 + Scaffold + 5 Membranen. Noch nichts pushen.

# bereits getan: git init -b main
$ git add -A
$ git commit -m "feat: initial scaffold — ADR-001/002, schema, 5 membranes"
✓ ADR-001 geschrieben ✓ ADR-002 geschrieben ✓ Scaffold lauffähig ✓ .gitignore drin
Warum jetzt: Erster Commit als sauberer Stand. Wenn später was schiefgeht, hast du einen Rollback-Punkt.
2
+0.5 Tag

GitHub-Repo erstellen, push (PRIVATE)

private remote

Repo auf GitHub anlegen, lokal pushen — aber noch private. Backup-Funktion, plus du kannst von anderen Geräten ran.

$ gh repo create pem-ec-designer --private --source=. --push
# oder via Web-UI dann:
$ git remote add origin git@github.com:USER/pem-ec-designer.git
$ git push -u origin main
✗ noch nicht öffentlich ✓ Backup gesichert
Warum private zuerst: README braucht noch Polish, License steht noch nicht (ADR-003 Late Binding). Public mit GPL-Konflikt + leerem README ist kein guter Eindruck.
3
Bei v0.1

CI einrichten (GitHub Actions)

noch private

Workflow für pytest + ruff + Schema-Drift-Check (auto-gen library/schema.json muss == committed sein). Ab jetzt jeder Push gegen Tests.

# .github/workflows/ci.yml
name: CI
on: [push, pull_request]
jobs:
  test:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-python@v5
        with: { python-version: '3.11' }
      - run: pip install -e ".[dev]"
      - run: pytest
      - run: ruff check src/ tests/
✓ Tests automatisch ✓ Schema-Drift erkannt ✓ Code-Quality-Gate
Warum jetzt: Sobald Tests existieren, sollten sie auf jedem Push laufen. Sonst driften Local-Branch und CI auseinander.
4
Bei v0.2

Public push

PUBLIC

Repo auf public umstellen. Voraussetzungen: README erklärt sich selbst, eine Demo-GIF zeigt was läuft, License ist gewählt (ADR-003), erste 3D-Komponente rendert.

$ gh repo edit --visibility public
$ git tag v0.2.0 && git push --tags
✓ Demo-GIF im README ✓ License (ADR-003) ✓ erste 3D-Komponente ✓ CI grün
Warum nicht früher: Erster öffentlicher Eindruck zählt. Ein leerer Designer ohne Visual ist Portfolio-schwach. Erst zeigen wenn was zu sehen ist.
5
v0.5 + v1.0

Releases mit Binaries

RELEASE

PyInstaller-Builds für macOS / Windows / Linux als GitHub Releases. CI baut auto bei Tag-Push. Industrie-Tool-Phase startet.

# .github/workflows/release.yml — bei tag-push v*
- run: pyinstaller pem_ec_designer.spec
- uses: softprops/action-gh-release@v2
  with:
    files: dist/*
✓ macOS DMG ✓ Windows MSI ✓ Linux AppImage
Warum erst spät: Cross-Platform-Builds sind eigene Wartungsarbeit. Nur sinnvoll wenn echte Nutzer das wollen. Bei v0.5 ist der Use-Case bewiesen.
